# Chrome Tab Accumulation Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** Eliminate Chrome tab accumulation by making all scripts reuse the existing x.com tab and preventing beforeunload dialogs from blocking navigation.

**Architecture:** Two root causes work together to accumulate tabs: (1) `open_selected_post()` creates a new post-URL tab every hourly run and never reclaims it; (2) every script's dialog-fallback calls `new_tab()` instead of navigating the current tab. A third, independent problem makes (2) worse: `goto_url()` to x.com can trigger a "leave page?" (beforeunload) dialog, which then causes the dialog-fallback to fire and create yet another tab. The fix addresses all three in one pass: expand tab-finding to prefer any x.com tab over creating fresh, suppress `window.onbeforeunload` before every `goto_url()` call, and replace all post-switch `new_tab()` fallbacks with `goto_url()`.

**Tech Stack:** Python f-strings/raw-strings embedding harness JS/Python snippets. No new files, no new `common.py` helpers. No test suite — verification is manual tab count observation.

---

## Background: Two classes of dialog, one fix

**Pre-existing dialog** (`page_info().get('dialog') == True` immediately after `switch_tab()`):
X.com JS has popped an alert/confirm, OR a previous navigation left a stuck beforeunload dialog on the tab. Current code bails out with `new_tab()`, leaving the stuck tab open. Fix: suppress beforeunload with JS then call `goto_url()` — CDP navigation clears most stuck dialog states.

**Navigation-triggered dialog** (beforeunload fires *during* `goto_url()`):
When a compose-box has content, x.com's `beforeunload` handler asks "leave page?". This already causes exceptions in the existing code (e.g. the unconditional `goto_url('https://x.com/home')` in `observe_feed.py`). Fix: run `js('window.onbeforeunload = null')` immediately before **every** `goto_url()` call.

**Standard navigation pattern** (used in every task below):
```python
js('window.onbeforeunload = null')
goto_url(target_url)
wait_for_load(20)
wait(N)
```

## Tab lifecycle analysis (background)

After a complete reply cycle:
- `collect_feed_posts()` → finds or creates `x.com/home` tab (reused ✓)
- `open_selected_post()` → creates **new** tab for post URL (e.g. `x.com/user/status/N`) — never closed ✗
- `send_reply.py` → finds and reuses the post-URL tab (reused ✓)

Each hourly run leaks one post-URL tab. After 24 hours: 24 open status tabs + home tab.

Dialog fallbacks in every script also create unclosed tabs: when `page_info().get('dialog')` is truthy after switching, the script calls `new_tab()` and abandons the previous tab.

**Steady-state goal after this fix:** at most 1 x.com tab open at any time, reused across all scripts in the same daemon session.

```python

for tab in tabs:
    tab_url = tab.get('url', '')
    if 'x.com/home' in tab_url:
        x_tab = tab
        break
if x_tab:
    switch_tab(x_tab['targetId'])
else:
    tid = new_tab('https://x.com/home')
    switch_tab(tid)
current = page_info()
if current.get('dialog'):
    tid = new_tab('https://x.com/home')
    switch_tab(tid)
    current = page_info()

    goto_url('https://x.com/home')
```

With:

```python
tabs = list_tabs(include_chrome=False)
x_tab = None
for tab in tabs:
    if 'x.com/home' in tab.get('url', ''):
        x_tab = tab
        break
if not x_tab:
    for tab in tabs:
        if 'x.com' in tab.get('url', ''):
            x_tab = tab
            break

    switch_tab(x_tab['targetId'])

    tid = new_tab('https://x.com/home')

    current = page_info()
```

Key changes:
- Second loop: fall back to any `x.com` tab before creating a fresh one
- Dialog fallback: `js('window.onbeforeunload = null')` then `goto_url()` instead of `new_tab()`
- Combined dialog + wrong-URL checks into one branch (same fix either way)
